Bài 1 : DevOps là gì và tại sao chúng ta sử dụng nó ?
1. Mục tiêu bài học
Sau bài này, người học cần hiểu được:
+ DevOps là gì, vì sao DevOps không chỉ là Docker, Kubernetes hay AWS.
+ Vì sao cần tự động hóa build, test, deploy, rollback và monitoring.
+ Các nhóm kiến thức nền tảng: Linux/WSL, Git/GitHub, networking, SSH, Bash Script, Docker, Nginx và CI/CD.
+ Cách một website đi từ code local đến GitHub Actions, Docker Hub, AWS EC2, Nginx, domain và HTTPS.
+ Các lệnh quan trọng thường gặp trong môi trường DevOps cơ bản.
+ Những lỗi người mới dễ mắc và cách tự kiểm tra khi gặp sự cố.
2. Khái niệm chính
2.1. DevOps là gì?
DevOps là cách kết hợp giữa phát triển phần mềm và vận hành hệ thống để phần mềm được build, test, deploy và theo dõi một cách tự động, ổn định và có kiểm soát.
Trong đó:
+ Development: viết code, phát triển tính năng, sửa lỗi.
+ Operations: vận hành server, cloud, mạng, bảo mật, log và hiệu năng.
+ DevOps: nối hai phần trên bằng quy trình, tự động hóa và công cụ phù hợp.
2.2. DevOps không chỉ là công cụ
Một hiểu lầm phổ biến là DevOps = Docker + Kubernetes + AWS.
Cách hiểu đúng hơn: DevOps = tư duy + quy trình + công cụ. Công cụ chỉ hỗ trợ; điều quan trọng là tổ chức quy trình làm phần mềm sao cho lặp lại được, ít lỗi hơn và dễ kiểm soát hơn.

2.4. Lộ trình học gợi ý
Người mới không nên học tất cả cùng lúc. Lộ trình hợp lý là đi từ nền tảng đến công cụ nâng cao:
+ Linux/WSL -> Git/GitHub -> Bash Script -> Networking -> SSH.
+ Docker -> Nginx -> CI/CD -> AWS EC2 -> Domain/HTTPS.
+ Monitoring -> Kubernetes -> Terraform.
Lưu ý: Không nên học Kubernetes quá sớm nếu chưa vững Linux, networking, Docker, CI/CD và cloud cơ bản. Nếu học quá sớm, người học dễ chỉ nhớ lệnh nhưng không hiểu bản chất.

3.2. Luồng tư duy của DevOps
+ Mọi thay đổi nên đi qua Git để có lịch sử và khả năng quay lại.
+ Build và test nên chạy tự động để phát hiện lỗi sớm.
+ Ứng dụng nên được đóng gói để môi trường chạy nhất quán.
+ Deploy nên có script hoặc pipeline để tránh thao tác thủ công lặp lại.
+ Sau deploy phải theo dõi log, tài nguyên và lỗi người dùng gặp.

BÀI 2: DÒNG CHẢY DỮ LIỆU (PIPE & REDIRECT) & CÔNG CỤ TÌM KIẾM THẦN THÁNH
Linux: "Mọi thứ trong Linux đều là File" và cách chúng ta điều hướng dòng chảy của dữ liệu “
1. Bản chất của Stream (Dòng dữ liệu)
Khi một lệnh Linux chạy, nó luôn mở ra 3 "đường ống" dữ liệu:
Stream
Số
Ý nghĩa
Stdin 
0
Dữ liệu đi vào 
Stdout 
1
Kết Quả Bình Thường
Stderr 
2
Thông Báo Lỗi
2. Kỹ thuật điều hướng (Redirect) và Kết nối (Pipe)
Kỹ thuật 1: Ghi đè (>) và Ghi nối đuôi (>>)
Thay vì in kết quả của một lệnh ra màn hình (stdout), ta có thể "bẻ lái" đường ống đó, bắt nó chui vào một file text để lưu lại.
> : Ghi đè. Nếu file đã có dữ liệu, nó sẽ xóa sạch đi và ghi mới.
Ví dụ: echo "Hello" > hello.txt (Tạo file hello.txt chứa chữ Hello).
>> : Ghi nối đuôi. Giữ lại dữ liệu cũ, ghi thêm vào cuối file.
Ví dụ: echo "World" >> hello.txt (File bây giờ sẽ có 2 dòng: Hello và World).
Ta dùng khi nào: Để gom log của ứng dụng. Ví dụ: chay_script.sh >> /var/log/my_app.log.
Kỹ thuật 2: Đường ống thần kỳ Pipe (|)
Dấu gạch đứng | (nằm trên phím Enter) dùng để lấy đầu ra (stdout) của lệnh phía trước làm đầu vào (stdin) cho lệnh phía sau. Nó cho phép em kết hợp nhiều lệnh đơn giản thành một bộ lọc cực kỳ mạnh mẽ.
+ Ví dụ sơ khai: Lệnh_A | Lệnh_B
3. Hai "Thám tử" tìm kiếm: grep và find
Lệnh 1: grep (Global Regular Expression Print)
- Dùng để làm gì: Tìm kiếm một từ khóa hoặc một chuỗi cụ thể bên trong file hoặc bên trong một luồng dữ liệu.
- Syntax: grep "từ_khóa" ten_file
- Options quan trọng:
-i (Ignore case): Tìm kiếm không phân biệt chữ hoa chữ thường.
-n (Line number): Hiển thị số dòng chứa từ khóa đó để dễ tìm.
ví dụ thực chiến với Pipe: Tưởng tượng file log của em dài 1 triệu dòng, em không thể mở nano lên đọc được. Em dùng Pipe kết hợp với grep:
Bash
cat /var/log/nginx/error.log | grep -i "critical"
Ý nghĩa: Lệnh cat sẽ đọc toàn bộ file log ra, nhưng thay vì xả 1 triệu dòng lên màn hình, dấu | sẽ chặn lại và đẩy đống chữ đó sang cho lệnh grep. Lệnh grep lọc đúng những dòng có chữ "critical" (bất kể hoa thường) để in ra cho em xem.
Lệnh 2: find
- Dùng để làm gì: Tìm kiếm vị trí của file hoặc thư mục dựa trên tên, kích thước, hoặc ngày tạo (Khác với grep là tìm chữ bên trong file).
- Syntax: find [nơi_muốn_tìm] -name "tên_file"
- Ví dụ: find /home -name "*.conf" (Tìm tất cả các file có đuôi .conf nằm trong thư mục /home).
Bài 3 : Lệnh Vim — Trình soạn thảo file trong Terminal
1. Vim là gì?
- Vim là trình soạn thảo văn bản chạy trong terminal Linux/WSL.
Dùng để sửa các file như:
vim note.txtvim deploy.shvim ~/.bashrcsudo vim /etc/nginx/nginx.conf
Trong DevOps, Vim hay dùng khi SSH vào server và cần sửa file cấu hình trực tiếp.
2. Điểm quan trọng nhất của Vim
- Vim khác Notepad/Nano vì nó có nhiều chế độ làm việc.
Có 3 chế độ cần nhớ:
Chế độ
Dùng để làm gì
Normal Mode
Di chuyển, xóa, copy, dán, chạy lệnh
Insert Mode
Gõ và sửa nội dung
Command Mode
Lưu, thoát, tìm kiếm, thay thế
Khi vừa mở Vim, mặc định bạn đang ở Normal Mode, nên chưa gõ chữ được ngay.
3. Quy trình dùng Vim cơ bản
Mở file → nhấn i → gõ nội dung → nhấn Esc → gõ :wq → Enter
Ví dụ:
vim note.txt
Trong Vim:

5. Script backup có log và xóa file cũ
#!/bin/bash
SOURCE_DIR="$HOME/devops-lab"
BACKUP_DIR="$HOME/backups"
LOG_FILE="$BACKUP_DIR/backup.log"
DATE=$(date +"%Y-%m-%d_%H-%M-%S")
BACKUP_FILE="devops-lab-$DATE.tar.gz"
KEEP_DAYS=7
mkdir -p "$BACKUP_DIR"
echo "===== Backup started at $(date) =====" >> "$LOG_FILE"
if [ ! -d "$SOURCE_DIR" ]; then
    echo "ERROR: Source directory not found: $SOURCE_DIR" >> "$LOG_FILE"
    exit 1
fi
tar -czf "$BACKUP_DIR/$BACKUP_FILE" "$SOURCE_DIR"
if [ $? -eq 0 ]; then
    echo "SUCCESS: $BACKUP_FILE" >> "$LOG_FILE"
else
    echo "ERROR: Backup failed" >> "$LOG_FILE"
    exit 1
fi
find "$BACKUP_DIR" -name "devops-lab-*.tar.gz" -type f -mtime +$KEEP_DAYS -delete
echo "===== Backup finished at $(date) =====" >> "$LOG_FILE"
echo "" >> "$LOG_FILE"

BÀI 5: QUẢN LÝ MÃ NGUỒN VỚI GIT & GITHUB (Công cụ quan trọng)
đã biết viết Script. Nhưng nếu vô tình em sửa nhầm làm hỏng file script đang chạy ngon lành, hoặc ổ cứng máy em bị hỏng thì sao? Đó là lý do Git ra đời.
1. Bản chất của Git là gì?
Hãy tưởng tượng Git giống như một "Cỗ máy thời gian". Nó chụp lại ảnh (Snapshot) toàn bộ mã nguồn của em tại một thời điểm. Nếu sau này em sửa code bị lỗi, em chỉ cần 1 lệnh là có thể "quay du hành thời gian" về trạng thái hoàn hảo trước đó.
Có 3 khu vực (Môi trường) trong Git mà em bắt buộc phải hiểu bản chất:
Working Directory (Thư mục làm việc): Nơi em đang gõ gõ sửa sửa file (thư mục devops_lab2 của em).
Staging Area (Sân chờ): Nơi em "nhặt" những file đã sửa xong xuôi, chuẩn bị đem đi lưu trữ. (Giống như em cho hàng vào giỏ siêu thị nhưng chưa ra quầy thanh toán).
Local Repository (Kho lưu trữ trên máy): Nơi Git chính thức đóng băng và lưu trữ lịch sử các file đó vào ổ cứng.
2. Các câu lệnh Git sinh tồn
Trước khi dùng Git, em phải khai báo "Danh tính" của mình với nó (chỉ cần làm 1 lần duy nhất trên máy):
Bash
git config --global user.name "Tên Của Em"
git config --global user.email "email_cua_em@gmail.com"
Command 1: git init
Dùng để làm gì: Biến một thư mục bình thường thành một "Kho chứa của Git" (Git Repository). Nó sẽ tạo ra một thư mục ẩn .git bên trong để âm thầm theo dõi mọi hành động của em.
Command 2: git status
Dùng để làm gì: "Hỏi thăm" xem Git đang thấy những file nào bị thay đổi, file nào chưa được lưu. DevOps luôn gõ lệnh này trước và sau khi làm bất cứ trò gì với Git.
Command 3: git add [ten_file]
Dùng để làm gì: Đưa file từ Thư mục làm việc vào Sân chờ (Staging Area).
Mẹo: git add . (Thêm TẤT CẢ các file có thay đổi vào sân chờ).
Command 4: git commit -m "Lời nhắn"
Dùng để làm gì: Chính thức bấm nút "Chụp ảnh lưu trữ" các file ở Sân chờ vào Kho lưu trữ. Lời nhắn (-m) phải ghi rõ em đã làm gì (Ví dụ: git commit -m "feat: add auto backup script").
BÀI 6 : COMPUTER NETWORKING - BẢN CHẤT CỦA MẠNG MÁY TÍNH (PHẦN 1)
1. Bản chất của IP Address và Port (Cổng)
Hãy tưởng tượng một Server (máy tính) giống như một Tòa chung cư.
- IP Address (Địa chỉ IP): Chính là Địa chỉ của tòa chung cư đó (Ví dụ: số 123 đường ABC). Nhờ có IP, các máy tính trên thế giới mới tìm thấy nhau.
- Port (Cổng): Chính là Số phòng trong tòa chung cư đó. Một máy tính có thể chạy cùng lúc nhiều phần mềm (Nginx, MySQL, SSH). Làm sao gói tin internet biết phải chui vào phần mềm nào? Nhờ vào số Port!
Ví dụ: Cổng 80 và 443 mặc định dành cho Web (Nginx). Cổng 22 dành cho kết nối từ xa (SSH). Cổng 5432 dành cho Database (PostgreSQL).

BÀI 7 : SSH - "CHÌA KHÓA VẠN NĂNG" ĐIỀU KHIỂN SERVER TỪ XA
Tưởng tượng sau này đi làm, công ty em thuê một cái Server vật lý đặt tại trung tâm dữ liệu ở Mỹ, hoặc mua Cloud của AWS. Em ngồi ở Việt Nam (hoặc đang ngồi quán cafe), làm sao em gõ được lệnh mkdir, nano, systemctl lên cái máy tính bên Mỹ đó?
1. Bản chất của SSH
- SSH là một giao thức mạng cho phép em mở một cửa sổ Terminal trên máy của em, nhưng mọi câu lệnh em gõ đều được mã hóa, truyền qua Internet, và chạy trên cái Server đích.
- SSH hoạt động theo mô hình Client - Server. Trên server đích phải chạy một Service tên là sshd (SSH Daemon), mặc định nó sẽ túc trực ở Cổng 22 (Port 22) để chờ kết nối vào.
2. Cú pháp kết nối SSH
Để chui vào một server khác, em cần biết 3 thứ: Tên User của server đó, Địa chỉ IP của nó, và Mật khẩu.
Syntax: ssh [tên_user_đích]@[địa_chỉ_IP_đích]
Ví dụ: ssh admin@192.168.1.100
Trên hệ thống Server thực tế, DevOps KHÔNG BAO GIỜ dùng mật khẩu để đăng nhập SSH. Mật khẩu có thể bị đoán mò (Brute-force attack) hoặc bị nhìn trộm. Chúng ta dùng SSH Keys (Chìa khóa mã hóa).
3. Bản chất của SSH Key (Ổ khóa và Chìa khóa)
Em sẽ tạo ra một cặp khóa đi liền với nhau:
- Private Key (Chìa khóa bí mật): File này em cất kỹ trong máy tính của em, không cho ai biết.
- Public Key (Ổ khóa công khai): File này em đem đi sao chép, ném lên mọi Server mà em muốn điều khiển (hoặc ném lên GitHub).
Cách hoạt động: Khi em gõ lệnh ssh kết nối vào Server, Server sẽ lấy cái Ổ khóa (Public Key) ra test. Nếu em có Chìa khóa (Private Key) khớp với ổ khóa đó, Server tự động mở cửa cho em vào ngay lập tức mà không cần hỏi mật khẩu. Cực kỳ an toàn và tiện lợi cho tự động hóa (Automation).
4. Các lệnh tạo và quản lý Key
Command 1: ssh-keygen
- Dùng để làm gì: Chế tạo ra cặp khóa (Public & Private Key).
- Cách dùng cơ bản: Gõ ssh-keygen rồi cứ thế ấn Enter liên tục cho đến khi xong (chấp nhận các giá trị mặc định).
- Kết quả: Nó sẽ sinh ra 2 file nằm giấu kỹ trong thư mục ẩn ~/.ssh/:
- id_rsa (Đây là Private Key - Tuyệt đối không gửi cho ai).
- id_rsa.pub (Đây là Public Key - Sẽ đem copy đi nơi khác).
Command 2: ssh-copy-id
- Dùng để làm gì: Copy cái Public Key của em nhét vào Server đích một cách tự động và an toàn.
- Syntax: ssh-copy-id [tên_user]@[đị_chỉ_IP_Server]

9. Docker Volume
9.1. Volume là gì?
Volume là nơi lưu dữ liệu bền vững cho container.
Mặc định, nếu dữ liệu nằm trong container và container bị xóa, dữ liệu có thể mất.
Volume giúp tách dữ liệu ra khỏi vòng đời container.
Ví dụ với MySQL:
docker volume create mysql_data
Chạy MySQL với volume:
docker run -d \
  --name mysql_db \
  -e MYSQL_ROOT_PASSWORD=123456 \
  -v mysql_data:/var/lib/mysql \
  mysql:8
Ý nghĩa:
Dữ liệu MySQL
↓
/var/lib/mysql trong container
↓
mysql_data volume trên Docker host
Container có thể bị xóa, nhưng volume vẫn còn.
9.2. Các lệnh volume quan trọng
Xem danh sách volume:
docker volume ls
Tạo volume:
docker volume create my_data
Xem chi tiết volume:
docker volume inspect my_data
Xóa volume:
docker volume rm my_data
Cảnh báo: Không xóa volume database nếu chưa backup.
10. Docker Network
10.1. Network là gì?
Docker Network giúp các container giao tiếp với nhau.
Ví dụ có 3 container:
frontend
backend
database
Nếu cùng nằm trong một Docker network, backend có thể gọi database bằng tên container:
database:3306
Không cần nhớ IP nội bộ của container.
10.2. Các lệnh network quan trọng
Xem danh sách network:
docker network ls
Tạo network:
docker network create app-network
Chạy container trong network:
docker run -d \
  --name backend \
  --network app-network \
  backend-image
Chạy database trong cùng network:
docker run -d \
  --name database \
  --network app-network \
  mysql:8
Backend có thể kết nối database bằng:
database:3306

  mysql_data:

  app-network:
Ý nghĩa:
web và mysql cùng nằm trong app-network.
mysql lưu dữ liệu vào mysql_data volume.

BÀI 9 : NGINX VÀ REVERSE PROXY
1. Nginx là gì?
Nginx (đọc là Engine-X) là một Web Server và Reverse Proxy Server hiệu năng cao, được sử dụng rộng rãi trong môi trường Production, DevOps và Cloud.
Nginx đóng vai trò là "người gác cổng" đứng trước ứng dụng, tiếp nhận các request từ người dùng và chuyển tiếp tới ứng dụng thực tế.
2. Kiến trúc không sử dụng Nginx

    ↓

    ↓

Nhược điểm:
+ Khó cấu hình HTTPS.
+ Khó mở rộng hệ thống.
+ Khó chạy nhiều ứng dụng trên cùng một máy chủ.
+ Không phù hợp với môi trường Production.
3. Kiến trúc sử dụng Nginx

    ↓

    ↓

    ↓

Nginx tiếp nhận request từ Internet và chuyển tiếp tới ứng dụng.
4. Reverse Proxy là gì?
Reverse Proxy là cơ chế trong đó Nginx đứng giữa người dùng và ứng dụng.
Ví dụ:
Người dùng truy cập:
https://app.trongdev.me
Nginx chuyển tiếp request tới:
http://127.0.0.1:8080
Người dùng chỉ biết đến domain mà không biết ứng dụng thực sự đang chạy ở đâu.

6. Ví dụ cấu hình Reverse Proxy
server {
    listen 443 ssl;
    server_name app.trongdev.me;
    ssl_certificate /etc/letsencrypt/live/app.trongdev.me/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/app.trongdev.me/privkey.pem;
    location / {
        proxy_pass http://127.0.0.1:8080;
    }
}

5. Chạy website bằng Docker trên EC2
Pull image từ Docker Hub:
sudo docker pull ndt2k7/trong-website:latest
Chạy container:
sudo docker run -d \
  --name my_web_container \
  -p 8080:80 \
  --restart unless-stopped \
  ndt2k7/trong-website:latest
Thành phần
Ý nghĩa
-d
Chạy container ở chế độ nền.
--name my_web_container
Đặt tên container.
-p 8080:80
Map port 8080 của EC2 vào port 80 trong container.
--restart unless-stopped
Tự chạy lại khi server restart.
Kiểm tra container:
sudo docker ps
curl http://localhost:8080
Nếu curl trả về HTML thì container đã chạy đúng.
6. Nginx Reverse Proxy và HTTPS
6.1. Vì sao Docker chạy port 8080?
Không nên để Docker chiếm port 80 vì Nginx cần dùng port 80 và 443 để nhận request HTTP/HTTPS từ người dùng.
Internet
↓
Nginx 80/443
↓
localhost:8080
↓
Docker Container 80
6.2. Cấu hình Nginx Reverse Proxy
Tạo file cấu hình:
sudo nano /etc/nginx/sites-available/app.trongdev.me
Cấu hình HTTP:
server {
    listen 80;
    server_name app.trongdev.me;
    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
Kích hoạt site và restart Nginx:
sudo ln -sf /etc/nginx/sites-available/app.trongdev.me /etc/nginx/sites-enabled/app.trongdev.me
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t
sudo systemctl restart nginx
6.3. Cấu hình HTTPS bằng Certbot
HTTPS giúp mã hóa dữ liệu giữa trình duyệt và server. Let’s Encrypt cấp SSL miễn phí, còn Certbot là công cụ xin và cài chứng chỉ SSL.
Cài Certbot:
sudo apt install certbot -y
Xin chứng chỉ SSL:
sudo certbot certonly --standalone -d app.trongdev.me
Nếu thành công, Certbot sẽ tạo certificate tại:
/etc/letsencrypt/live/app.trongdev.me/fullchain.pem
/etc/letsencrypt/live/app.trongdev.me/privkey.pem
Lưu ý quan trọng
Certbot phải chạy trên EC2 thật, domain phải trỏ đúng về EC2, port 80 phải mở trong Security Group. Không chạy Certbot trong WSL local nếu domain đang trỏ về EC2.
6.4. Cấu hình Nginx HTTPS hoàn chỉnh
server {
    listen 80;
    server_name app.trongdev.me;
    location / {
        return 301 https://$host$request_uri;
    }
}
server {
    listen 443 ssl;
    server_name app.trongdev.me;
    ssl_certificate /etc/letsencrypt/live/app.trongdev.me/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/app.trongdev.me/privkey.pem;
    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
Kiểm tra và restart Nginx:
sudo nginx -t
sudo systemctl restart nginx
sudo ss -tulpn | grep -E ':80|:443|:8080' 
7. GitHub Secrets và file ci.yml
7.1. GitHub Secrets cần tạo
Không được ghi password, token hoặc private key trực tiếp vào code. Cần tạo các GitHub Secrets sau:
Secret name
Ý nghĩa
DOCKERHUB_USERNAME
Username Docker Hub.
DOCKERHUB_TOKEN
Token Docker Hub.
SERVER_HOST
Domain hoặc IP của EC2.
SERVER_USERNAME
User SSH của EC2.
SERVER_SSH_KEY
Private key dùng để SSH.
Nguyên tắc bảo mật
Không commit password, Docker Hub token hoặc SSH private key lên GitHub. Tất cả thông tin nhạy cảm phải đưa vào GitHub Secrets.
7.2. File GitHub Actions CI/CD
Tạo file:
.github/workflows/ci.yml
Nội dung mẫu:
name: Build and Deploy Docker App
on:
  push:
    branches:
      - main
env:
  IMAGE_NAME: ${{ secrets.DOCKERHUB_USERNAME }}/trong-website:latest
  CONTAINER_NAME: my_web_container
  HOST_PORT: 8080
  CONTAINER_PORT: 80
jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout source code
        uses: actions/checkout@v4
      - name: Login to Docker Hub
        run: |
          echo "${{ secrets.DOCKERHUB_TOKEN }}" | docker login \
            -u "${{ secrets.DOCKERHUB_USERNAME }}" \
            --password-stdin
      - name: Build Docker image
        run: |
          docker build -t $IMAGE_NAME .
      - name: Push Docker image to Docker Hub
        run: |
          docker push $IMAGE_NAME
      - name: Prepare SSH key
        run: |
          mkdir -p ~/.ssh
          echo "${{ secrets.SERVER_SSH_KEY }}" > ~/.ssh/id_rsa
          chmod 600 ~/.ssh/id_rsa
          ssh-keyscan -H "${{ secrets.SERVER_HOST }}" >> ~/.ssh/known_hosts
      - name: Deploy to AWS EC2
        run: |
          ssh "${{ secrets.SERVER_USERNAME }}@${{ secrets.SERVER_HOST }}" << EOF
            docker pull $IMAGE_NAME
            docker stop $CONTAINER_NAME || true
            docker rm $CONTAINER_NAME || true
            docker run -d \
              --name $CONTAINER_NAME \
              -p $HOST_PORT:$CONTAINER_PORT \
              --restart unless-stopped \
              $IMAGE_NAME
            docker ps
          EOF

10. Lỗi thường gặp
Lỗi
Nguyên nhân thường gặp
Cách kiểm tra
Workflow không chạy
File ci.yml đặt sai vị trí hoặc không push lên main.
Kiểm tra .github/workflows/ci.yml.
Docker login failed
Sai Docker Hub username/token.
Kiểm tra GitHub Secrets.
Pull access denied
Sai tên image hoặc image private.
Kiểm tra IMAGE_NAME.
SSH vào EC2 thất bại
Sai host, user hoặc private key.
Kiểm tra SERVER_HOST, SERVER_USERNAME, SERVER_SSH_KEY.
Permission denied publickey
Private key không khớp với public key trên EC2.
Kiểm tra ~/.ssh/authorized_keys.
Docker permission denied
User chưa có quyền Docker.
Dùng sudo docker hoặc thêm user vào group docker.
Port is already allocated
Port đang bị process khác chiếm.
Kiểm tra sudo ss -tulpn.
Website không vào được
Nginx, Docker, DNS hoặc Security Group lỗi.
Kiểm tra từng lớp.
502 Bad Gateway
Nginx không kết nối được container.
Kiểm tra curl localhost:8080.
HTTPS lỗi
Sai certificate hoặc port 443 chưa mở.
Kiểm tra Nginx config và Security Group.
